1. Project Overview
LearnSphere is a full-stack AI-powered academic intelligence platform. It was initially built as a personal study assistant (v1) and is being extended into an institutional at-risk student detection and intervention system (v2).

4. v2 Architecture Overview
v2 adds an institutional intelligence layer on top of v1. All v1 features remain unchanged. v2 adds risk scoring, intervention pipelines, and 5 new role-based dashboards.

4.1 System Architecture
Data Layer (Student activity + LMS ingestion + Attendance) → Risk Score Engine (Rule layer + AI explanation) → Alert & Triage System → Intervention Pipeline (Educator → HoD → Student) → Outcome Tracking (7-day follow-up)

4.2 Six User Roles
Role
Scope
Key Power
STUDENT
Themselves only
Learn, track progress, receive interventions, see own risk score
PARENT
Their 2 children max (read-only)
View child risk/attendance/marks, receive RED+AMBER alerts, chat with educators
EDUCATOR
Their subject's students
Create intervention plans, view student risk per subject
ADVISOR
Assigned students only
Log counselling notes, track intervention outcomes, full student view
HOD
Full department
Risk heatmap, NAAC reports, educator oversight, set risk thresholds, approve RED interventions
ADMIN
Entire institution
All user management, bulk CSV import, LMS config, audit logs, system health

4.3 Intervention Approval Flow
This is the exact sequence for a RED-level student. No shortcuts.
AI risk engine detects student hits RED (score > 60)
Educator creates intervention plan (can use AI suggestion as draft)
HoD reviews — approves or modifies the plan
Plan sent to student dashboard
Parent receives alert (in-app + email via nodemailer)
Advisor logs counselling notes if meeting occurs
7 days later: cron job recalculates risk score, logs outcome delta

4.4 Login Flow
Single login page at /login. Role selector dropdown (STUDENT / PARENT / EDUCATOR / ADVISOR / HOD / ADMIN). JWT returned from server contains { userId, role, email }. RoleRouter.tsx reads role and redirects to the correct portal on login.

6. Risk Score Engine — Full Specification
IMPORTANT: The risk engine is hybrid. The RULE LAYER calculates all scores deterministically. Gemini ONLY writes the explanation and suggests interventions. Gemini never decides the score.

6.2 Scoring Logic
Each dimension produces a safety score (0-100, higher = safer). These are weighted and summed into a composite safety score. Risk score = 100 - safetyScore. Risk score is then bucketed:
0-30 → GREEN (On track)
31-60 → AMBER (Needs attention)
61-100 → RED (Urgent intervention required)

6.4 AI Layer — Gemini Constraints
The AI layer receives: composite score, risk level, all dimension scores, all active flags with their detail strings, and student name. It is constrained by:
System instruction: ONLY reference data provided — no invented context
responseMimeType: "application/json" — forces JSON-only output
temperature: 0.1 — minimises hallucination
Zod validation on output before anything touches DB or UI
Fallback rule-based explanation if AI fails or key not configured

AI must return this exact JSON structure:
{ explanation: string, predictedTrajectory: string, interventions: [{ rank: 1|2|3, title: string, action: string, urgency: "immediate"|"this_week"|"this_month", targetDimension: "attendance"|"marks"|"study_activity"|"lms_activity"|"recall"|"behavioural" }] }

12. CSV Import Formats

12.1 Bulk Student Import (Admin)
Use Option B — Full format. This creates users AND automatically links parents and assigns advisors in one upload.
Filename: students_import.csv
Headers (row 1):
email,password,name,role,departmentId,rollNumber,year,section,parentEmail1,parentEmail2,advisorEmail
Notes:
role must be one of: STUDENT, PARENT, EDUCATOR, ADVISOR, HOD, ADMIN
parentEmail1 and parentEmail2 are optional — if provided, a ParentStudentLink is auto-created
advisorEmail is optional — if provided, an AdvisorAssignment is auto-created
departmentId must match an existing Department id in the database

12.2 Attendance Import
Headers: userId,subjectId,date,status
date format: YYYY-MM-DD. status: PRESENT, ABSENT, LATE, EXCUSED

12.3 LMS Activity Import
Headers: userId,activityType,durationMins,loggedAt
loggedAt format: ISO 8601. activityType examples: login, assignment_submit, resource_view, forum_post

13. WebSocket Chat — Technical Spec

13.1 Room Structure
Rooms are scoped by studentId: chat:{studentId}
Members of a room: all parents of that student + all educators of that student's subjects + all advisors assigned to that student

13.2 Events
Direction
Event
Payload
client → server
chat:join
{ studentId, userId, role }
client → server
chat:message
{ studentId, toId, message }
client → server
chat:typing
{ studentId, userId, isTyping }
server → client
chat:message
{ id, fromId, fromName, message, createdAt }
server → client
chat:history
{ messages: ChatMessage[] }
server → client
chat:typing
{ userId, isTyping }
server → client
chat:read
{ messageIds: string[] }

13.3 Persistence
All messages are persisted to the ChatMessage table in the database. On chat:join, the server sends the last 50 messages as chat:history. This ensures chat history survives page refresh and reconnects.

14. NAAC/NBA Report — PDF Specification

Generated server-side using pdfkit. Endpoint: POST /api/hod/reports/naac
Language: Trilingual — English headers with Tamil and Hindi translations below each section title.

14.1 Report Sections
Executive Summary — institution name, date range, total students, RED/AMBER/GREEN counts, overall intervention rate
Department Risk Overview — table: department name, student count, RED count, AMBER count, avg risk score
Subject-wise Failure Rate Heatmap — table: subject name, enrolled count, below 40% count, below 50% count, avg marks %
At-Risk Student Remediation Evidence — anonymised list: student ID, risk level, intervention sent date, outcome (improved/pending/worsened)
Educator Engagement Metrics — educator name, interventions created, avg response time (hours), completion rate
Trend Analysis — risk score distribution over selected date range. Text summary of improvement or deterioration.
Appendix — methodology note explaining the hybrid risk scoring approach
